## Detailed Bio (200 words)

Nikita Filatov is a software engineer and open source maintainer specializing in developer tools for the AI era. Based in Berlin, he works at the intersection of systems programming and AI-assisted development.

The inspiration for ws came from firsthand experience: while pair-programming with Claude Code and Cursor, Nikita found himself constantly losing track of which files were relevant to his current feature. AI assistants could identify relevant files, but there was no persistent way to remember them across sessions or branches.

He built ws to solve this problem: a terminal UI tool that maintains branch-scoped working sets, giving AI assistants focused context and developers an easier way to navigate their codebase. Released as open source in 2025, ws now integrates with Claude Code, Cursor, GitHub Copilot, and popular editors.

Nikita's work focuses on making AI assistants more effective by giving them better context. He believes that the future of programming is human-AI collaboration, and builds tools to make that collaboration seamless.

Previously, Nikita worked on distributed systems and developer tooling at various startups. He contributes to Go and open source tools, and writes about AI-assisted development workflows.

## Talking Points

### On AI Pair Programming
- "AI assistants are great at discovery but terrible at persistence. ws gives them that persistence."
- "The future of programming isn't AI replacing developers — it's AI + developers collaborating better. ws is a tool for that collaboration."
- "When your AI assistant knows your working set, it can provide focused, relevant answers instead of generic ones."

### On Branch-Scoped Context
- "Every feature branch should have its own context. Switching branches should switch your working set automatically."
- "Manual context management is a distraction. Make it automatic and branch-scoped."
- "ws is the missing primitive: branch-scoped file lists for git projects."

### On Developer Tools
- "Build tools for the way you actually work, not the way you think you should work."
- "Terminal-first tools work with any editor. Don't lock users into one ecosystem."
- "The best tools are invisible. They just work and get out of the way."

### On Open Source
- "ws is open source because developer tools should be community-driven."
- "I built ws for my own workflow, but released it because others might find it useful."
- "The best tools are forged in real-world use, not product meetings."
